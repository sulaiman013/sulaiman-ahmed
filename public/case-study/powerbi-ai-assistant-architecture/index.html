How it answers

Three answer paths, one endpoint

The BOT routes to one of three answer paths based on the shape of the question. This is not a RAG chatbot — the "knowledge" isn't retrieved text; it's a query result, an image interpretation, or a curated guide block.

~70%

DAX agent

numerical questions

Either a hand-curated recipe (deterministic DAX) OR an LLM that picks the measure + writes DAX, calling query_dax against the Power BI semantic model. Returns real numbers.

Not RAG — the answer is a query result.
~20%

Vision path

screenshot questions

Vision LLM reads the captured image of the current report page. Can optionally call query_dax if the user asks "why" or "vs last year".

True multimodal interpretation, not OCR.
~10%

Guide path

"where do I find X?"

The full report guide (~1500 tokens) is eagerly injected into the system prompt. The LLM extracts the relevant section.

Not vector retrieval — small enough to inject the whole thing.
Mental model

The BOT is an onion

Each layer is independently swappable. Layers 0–4 lift verbatim across clients; layers 5–7 are domain-specific.

01 Architecture

Two paths through one endpoint

Every request hits a single Edge route. A pre-LLM matcher decides between the deterministic fast path and the LLM-driven slow path. Streaming is non-negotiable — the user sees the first token in <2 s.

The main difference in one line

The slow path is generative. The AI itself writes the DAX query from scratch every time. The fast path is lookup. A pre-LLM matcher finds a hand-curated recipe and runs it directly. That single decision — who writes the query — cascades into everything else.

Dimension Slow path Fast path
Who writes the DAX The AI, fresh every time Pre-written recipe
Cost per call ~$0.0014 (post-opt) · up to $0.018 with image ~$0.00018 — 50× cheaper
Latency 8 – 25 seconds 2.5 – 5 seconds — 3× faster
Determinism Same question may produce slightly different DAX Same question → same DAX → same number, always
Coverage Anything in the semantic model Only the question shapes you pre-wrote
Production hit rate ~70% of calls (long tail, paraphrases, image questions) 27.6% of calls (top-frequency questions)

You need both. The fast path catches the questions owners notice when answers vary — the highest-frequency 30% where determinism is non-negotiable. The slow path catches everything else, including new shapes you have not pre-authored. The recipe layer sits in front of the slow path: every request tries the matcher first, falls through to the LLM-driven path on a miss.

The real reason to ship the fast path is not cost. It is trust. The 50× cost reduction is a bonus.

L1 — Sign-in. Supabase email + magic-link auth. Every request to the BOT route revalidates the JWT against the auth server.
L3–L4 — Active branch + dataset routing. Top-left shows Demo Branch; top-right shows "Scoped to your branch". Every PBI query in this session is pinned to this branch's dataset.
Why service principal, not per-user PBI tokens
  • Per-user tokens require a PPU license per user (~$10/user/month). IC has hundreds of franchise users; only ~10 PPU licenses.
  • Service-principal token refresh is deterministic (1-hour TTL); per-user is on inactivity.
  • SP needs Member on the workspace — not Viewer. (Viewer can't run executeQueries.)
Pitfalls
  • branch_code is text, not enum — TS-only safety. Wrong spelling silently passes through.
  • Service-role vs anon key — always anon for the BOT route; SR is for migrations only.
  • Branch switch must reset chat — otherwise ICAU history flows into ICLA conversation.
  • Embed URLs are server-only — leaking them gives the entire dashboard to anyone.

"Summarize this view" in production — the user clicks the canned button, the browser captures the full Pipeline page as a JPEG (quality 0.92, max 1920px), the vision model reads it, and the wrap-up streams a clean five-bullet summary. The screenshot is stripped from the conversation history before any subsequent tool round.
Why JPEG 0.92 (not 0.82)
Power BI renders charts with thin lines and small numbers. At quality 0.82 the vision model misread digits ("$74,318" → "$74,910"). Bumping to 0.92 fixed it. The tradeoff: image tokens are expensive — a 1920px screenshot is 5–15K tokens at GLM's $0.60/M = ~$0.003–$0.009 just for the image, before any text processing. This is the #1 reason we strip the image after round 0.

Live DAX in action. User asks "who are my top 3 designers?" → vision pass detects no image is needed → DAX writer emits a TOPN(3, ...) against the active branch's dataset → wrap-up renders the result. End-to-end: ~6 s. The model never invented these names — they came from Salesforce_Data[Designer] via executeQueries.
Why errors are strings, not exceptions

If executeDax threw on every PBI failure, the route would need try/catch around every dispatch. Instead it returns {error: "..."} and the formatter turns that into a string the model sees verbatim. The model recovers — "oh, that column doesn't exist, let me look at the SEMANTIC MODEL section" — and writes a corrected query in round 1 or 2. This is exactly the OpenAI tool-use error-handling pattern.

The BLANK detection

A CALCULATE([m], filter) whose filter matches zero rows returns one row of all-null columns — looks like success but means "filter eliminated everything." We surface this explicitly with a check-list ("spelling? date range? broaden first then narrow?"). Without it the BOT would tell users "your revenue is $null" — wrong AND misleading.

Why we swapped GLM-4.5V → Gemini 2.5 Flash (2026-05-05)

Production audit of 332 rows surfaced two GLM-4.5V failure classes:

  1. XML-wrapper tool-call leak — emitted Qwen-style markup as plain text instead of OpenAI tool_calls.
  2. Empty arguments strings — wasted a tool round before retry. ~12 of 14 DAX-using calls in the sample.

Both are GLM-specific. Gemini 2.5 Flash uses the same OpenAI-compatible tool-call format as our DAX writer → these failure classes literally cannot occur. Bonus: 50% cheaper input rate.

2026-05-05 reliability layer

Four server-side defenses, none model-specific (defense-in-depth):

  • XML tool-call recovery — 250-char probe extracts DAX from wrapper
  • Numerical hallucination guard — re-checks bare-numbers-no-DAX answers
  • Server-side {YEAR}/{MONTH} substitution — placeholder leaks no longer 400
  • Empty-DAX early reject — tool description forbids; minLength: 10 in JSON schema

Spec-vs-model drift caught 2026-05-03
Exploration script _explore_dax_patterns.mts probed the live ICAU model via INFO.VIEW.COLUMNS() and found 9 columns the BOT thought existed didn't actually exist in the M-partition (geographic columns, install dates, etc). All 9 removed; comment block now points readers to "geographic data lives on Leads, not Salesforce_Data." Run that probe monthly to catch future drift.

Per-branch real-data context

Schema knowledge tells the model what columns exist. It does NOT tell the model what values those columns contain. That's how the BOT learns that "Alyssa" means "Alyssa Jordan" in ICAU's roster — without round-tripping a DAX query.

The roster paying off. The user wrote "top 3 designers" — no full name given. The branch snapshot, injected into the system prompt as ~600-800 tokens of recognized names, told the model that Alyssa Jordan, Beth Cloutier, and Catherine Meyer are real designers in this branch's roster. The DAX query then ranked them on live data.
"How is Alyssa doing?" — fixed in one round

The model's prompt context already contains "Alyssa Jordan (1,247 projects, $812K)". It writes:

EVALUATE
ROW("Sold", CALCULATE(
  [Revenue Sold],
  Salesforce_Data[Designer] = "Alyssa Jordan",
  YEAR(Salesforce_Data[Created Date]) = 2026
))

Right name, first try. Without the snapshot: 1-2 disambiguation rounds, $0.001 + 3s wasted.

Why rounded numbers ($812K not $812,403)

Precise dollars in the prompt would tempt the model to quote them as live values. Pre-rounded values are visually obvious as stale snapshot data. The model has been observed to round less when it sees exact numbers, more when it sees pre-rounded ones. Subtle but real.

Why the BOT answers FROM the guide, not by linking TO it
The user is already in the BOT. Bouncing them to another surface for a definition is bad UX. The BOT's whole value-prop is "answer questions immediately." The system-prompt token cost (~$0.0004) is the price of avoiding the redirect.

The SSE buffer-pop pattern is mandatory
A single reader.read() chunk can split a JSON line in half. If you naively JSON.parse each split("\n") element, ~1 in 100 calls will throw. The pop-the-incomplete-tail pattern handles it. The tool-call delta indexing is by tool-call position, not token order — two parallel tool calls' arguments interleave on the wire; you need the index field to demultiplex.

The four high-leverage prompt patterns IC discovered
Pattern A
TIME SCOPE HARD RULE

LLMs LOVE adding YEAR(...)=N filters when not asked. Write the rule explicitly + 8–10 real failure examples with corrected DAX. The LLM internalizes from concrete examples better than abstract rules.

Pattern B
MEASURE PICKER table

For each pair of similarly-named measures the LLM confuses, add a numbered row: USE: ... / NOT: .... Numbered list, not paragraph form — LLM treats each as a discrete rule.

Pattern C
OUTPUT SHAPE rules

The wrap-up model needs explicit shape guidance: list rows when user asked to LIST/SHOW; 1-2 sentences for scalar; 2-4 sentences calling out top 2-3 for multi-row. Without this it picks shape randomly.

Pattern D
empty=empty scorer rule

When CSV expected reads "(no rows)" and BOT correctly says "couldn't find" — that's a SEMANTIC MATCH, not no_data. Recovered +5 questions in IC's Phase 8 alone.

EVAL

Two scorers, in parallel

Single-scorer evaluation is misleading. You need both — they answer different questions.

SCORER A
Answer-based
"Did the user see the right number?"
  • What it does — extract first numeric token, compare ±0.5% / ±5%
  • Tiers — exact · close · different · no_dax · no_data
  • Good at — UX proxy, spotting major regressions
  • Bad at — data drift, multi-row first-token brittleness, equivalent measures
  • Latest — 91.8% (156/170)
SCORER B
DAX-similarity
"Did the BOT reason correctly?"
  • What it does — compare structural features: measures, columns, TOPN/sort, equivalence map
  • Tiers — MATCH · PARTIAL · WRONG_MEASURE · NO_DAX
  • Good at — data-drift-immune, regression signal, per-failure diagnostics
  • Bad at — same shape ≠ right answer, requires reference DAX
  • Latest — 94.1% (160/170)
When the two disagree, DAX-similarity is the more meaningful number for evaluating CHANGES
Use answer-similarity to evaluate USER EXPERIENCE. After IC's Phase 6 they decoupled: DAX moved 90% → 94%, answer stayed 81–85%. The gap was scorer artifacts (decimal-vs-percent format, multi-row first-token brittleness, ICAU-specific entities).

Postmortems & silent-failure modes

Real bugs that shipped to production, the silent symptom, and the observability that would have caught them earlier. Build CI for these from day one if you can.

2026-05-06
Recipe matcher silent failures

Symptom: two latent matcher bugs silently degraded accuracy by ~5pp for weeks. Both presented identically: recipe_id=NULL, matcher_method=NULL, error=NULL.

Cause 1: optional named-group spread overwrites defaults. (?<N>\d+)? sets m.groups.N = undefined, spread {...defaults, ...m.groups} destroys the default.

Cause 2: missing context placeholders not in defaults or contextPlaceholders().

Fix: migration 010 added recipe_skip_reason + intent_classes. Bugs visible in 30s of SQL after that.

2026-05-07
Demo project notes NULL

Symptom: demo data generator produced rows with None in text fields because Faker doesn't have domain text generators.

Fix: pre-launch visual smoke test — display surfaces flagged blank rows during demo prep.

Lesson: don't trust generators silently; always run a 1-row visual check before demos.
